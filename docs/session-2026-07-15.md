# Session 2026-07-15/16 — Full DEUS v3 Overhaul

## Issues & Resolutions

### 1. Gmail API scope mismatch (`gmail.send` → `gmail.modify`)
- Old token had `gmail.send` scope, code requested `gmail.modify`.
- Token refresh failed with `invalid_scope: Bad Request`.
- **Fix:** Updated scope in `gmail_sender.py:54` and `app/api/routes.py`.

### 2. GmailSender singleton cached `available=False` permanently
- `GmailSender` initialized at app startup before any OAuth token existed.
- `_init_service()` set `self._available = False` and `get_gmail_sender()` returned the stale singleton forever.
- Even after successful OAuth callback saved a new token to DB, the singleton never rechecked.
- **Fix:** `gmail_sender.py:211` — `get_gmail_sender()` now creates a fresh instance if `available` is `False`.

### 3. Invalid Gemini API key
- `GEMINI_API_KEY` env var contained an expired/invalid key.
- All Gemini calls returned `API key not valid`.
- **Fix:** Updated to new key in `.env`. User must also update on Railway.

### 4. Invalid Groq API key (removed)
- `GROQ_API_KEY` env var contained an invalid key.
- `outreach_agent.py` attempted Groq as fallback after Gemini failed.
- **Fix:** Removed `GROQ_API_KEY` from Railway env vars. Code checks `if groq_key:` before using.

### 5. SMTP blocked on Railway
- Outbound SMTP ports (465, 587) blocked by Railway's network policy.
- `[Errno 101] Network is unreachable` on both ports.
- **Resolution:** Gmail API (HTTP-based) is the primary send method.

### 6. No Resend API key configured
- Resend was the last fallback in `email_sender.py` but no key was set.
- **Not an issue** — Gmail API is sufficient.

### 7. Spam trigger words in default subject
- `outreach_config.py` default subject: "Quick proposal for {business_name}"
- SpamChecker weights: "proposal" (+3), "quick call" (+2)
- **Fix:** Changed to "Quick opportunity for {business_name}"

### 8. 6 of 7 leads skipped in prior batch
- Caused by no batch delay between sends, rate limiter hourly cap, misleading Gmail cap redirect.
- **Fix:** Added batch delay in `outreach_agent.py`, removed misleading redirect.

### 9. Daemon auto-scout sent to scout leads during manual send
- Daemon's `_auto_outreach_scout()` ran every 4 minutes independently, sending to scout leads even while user manually sent to imported leads.
- **Fix:** Added `manual_send_active` flag in `outreach_agent.py`. Daemon checks this flag and skips auto-scout if manual send is running.

### 10. No progress indicators for long-running operations
- Sending emails and scouting for leads had no visual feedback on duration.
- **Fix:** Added progress bars with real-time polling:
  - Send progress bar (polls `/api/outreach/status/{job_id}` every 2s)
  - Scout progress bar (polls `/api/lead-scout/status/{job_id}` every 1.5s)
  - Both show percentage, current item, and counts.

### 11. Daemon auto-started on boot (undesired)
- Server boot triggered daemon start immediately, sending emails before user could intervene.
- **Fix:** Changed `DAEMON_AUTO_START` default from `"true"` to `"false"` in `app/main.py`.

### 12. No 24-hour auto-stop for daemon
- User wanted daemon to run for exactly 24 hours then stop automatically.
- **Fix:** Added `start(auto_stop_hours=24)` method to `DeusDaemon`. Timer thread calls `stop()` after 24h. Status shows remaining hours. New `/api/daemon/start-auto` endpoint. "Auto-Start (24h)" button in dashboard.

### 13. Testing/production mode toggle (unnecessary complexity)
- `mode_config.py` had full toggle infrastructure with state file, env var override, UI button.
- User wanted to remove testing mode entirely — always production.
- **Fix:** Simplified `mode_config.py` to hardcoded production config (90-180s delays, 100/day, 15/hr). Removed `/api/mode` endpoints. Removed mode toggle button from dashboard header.

### 14. Weekly email statistics not tracked
- No visibility into daily/weekly email send counts per method.
- **Fix:** Created `email_stats.py` — tracks daily counts per method (Gmail API/SMTP/Resend), rolling 7-day window, auto-prunes old entries. Hooks into `SendLimiter.record_send()`. Added `/api/email/stats/weekly` endpoint. Dashboard shows weekly stats table in Deliverability panel.

### 15. Leads tab missing from dashboard
- No way to view all leads in one place segmented by origin.
- **Fix:** Added "Leads" nav item and panel in dashboard. `/api/leads/segmented` endpoint returns leads grouped by `lead_type` (imported/scraped/other) with dates. Falls back to `leads.json` if DB table empty or column missing.

### 16. Leads panel showed blank due to CSS specificity bug
- Added `style="display:none"` inline on Leads panel div, which overrode `.panel.active{display:block}` CSS class.
- **Fix:** Removed inline `style` — panel now relies on CSS class toggling like all other panels.
